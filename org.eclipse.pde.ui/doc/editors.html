Resource Content Editors

Identifier

org.eclipse.ui.editors

Description

This extension point allows tools to register editors for resources. When the Open action is performed on a resource in theNavigator, the workbench registry is consulted to locate the registered editor for its type. The actual activation depends on the type of the editor. Embedded editors are tightly integrated into the workbench. In contrast, external editors are launched in a separate frame window with various level of integration with the workbench.

An embedded editor is in fact a combination of a number of framework class implementations, some required, some optional. Collectively, these make up "the tool" that is activated when the appropriate resource is selected and the appropriate action is triggered. Embedded editors can be implemented in Java or by using ActiveX components. In the case of Java, tight integration is achieved by using the same classes as the workbench itself. In the case of ActiveX, integration is achieved by providing an ActiveX container for the component and by allowing the component to merge into the menu bar and toolbar using OLE merging algorithms.

Internal editors written in Java are allowed to contribute to workbench menu bar, toolbar and status line. This contribution is achieved by registering a contributor class. Only one instance of this class is created per editor type, regardless of the number of editor instances in the workbench. The currently active editor instance is passed to the contributor using the specific API.

Markup

   <!ELEMENT extension (editor*)>
   <!ATTLIST extension
     point CDATA #REQUIRED
     id    CDATA #IMPLIED
     name  CDATA #IMPLIED
   >

  • point - a unique extension point identifier
  • id - an optional extension identifier
  • name - an optional extension name

   <!ELEMENT editor (EMPTY)>
   <!ATTLIST editor
     id               CDATA #REQUIRED
     name             CDATA #REQUIRED
     icon             CDATA #IMPLIED
     extensions       CDATA #REQUIRED
     extendedType     CDATA #IMPLIED
     class            CDATA #IMPLIED
     command          CDATA #IMPLIED
     contributorClass CDATA #IMPLIED
     default          (true | false) "false""false"
   >

  • id - a unique name that will be used to identify this editor
  • name - a translatable name that will be used in the UI for this editor
  • icon - a relative name of the icon that will be used for all resources that match the specified extensions. An icon is not required if you specify a command rather than a class. In that case, the workbench will use the icon provided by the operating system.
  • extensions - one or more tokens that will be used to recognize the type in the resource name. A resource type consists of its extension and an optional extended type. This allows definition of several types that have the same extension, but different extended types (for example, registering different editors for different classes of XML files while still keeping the familiar *.xml extension). By specifying multiple extensions, it is possible to account for variations that can be found in the real world (for example, both "htm" and "html" are valid for hypertext documents).
  • extendedType - an optional attribute that can be used to assign different editors for files of the same extension (for example, XML files that should be edited using different editors). Either extension, extendedType or both must be specified.
  • class - a name of the class that implements org.eclipse.ui.IEditorPart. This attribute may be left undefined if this is an external editor.
  • command - a command to run in order to launch an external editor. Attributes class and command are mutually exclusive.
  • contributorClass - a name of the class that implements com.ibm.itp.ui.api.contributions.IEditorActionBarContributor. This class provides for contributions to the workbench menu bar and toolbar when the editor is active. Only one instance of the contributor is made per editor ID, regardless of the number of editor instances currently opened.
  • default - if true, this editor will be used as the default editor for the type. This is only relevant in a case where more than one editor is registered for the same type. If an editor is not the default for the type, it can still be launched using "Open with..." submenu for the selected resource.

Example

The following is an example of an editor extension definition: 

   <extension point="org.eclipse.ui.editors"> 
      <editor 
         id="com.xyz.XMLEditor" 
         name="Fancy XYZ XML editor" 
         icon="./icons/XMLEditor.gif" 
         extensions="xml" 
         class="com.xyz.XMLEditor" 
         contributorClass="com.xyz.XMLEditorContributor" 
         default="false"> 
      </editor> 
   </extension>

API Information

If the attribute command is used, it will be treated as an external program command line that will be executed in a platform-dependent manner. If the attribute class is used, the workbench will assume that it is an internal editor and the specified class must implement org.eclipse.ui.IEditorPart. However, it is common practice to extend org.eclipse.ui.EditorPart in order to inherit the default functionality.

If attribute contributorClass is specified, it must represent a name of the class that implements com.ibm.itp.ui.api.contributions.IEditorActionBarContributor. This interface provides for adding menus and actions into the workbench menu bar and toolbar using the following legal paths:

  • edit.copy - for additional flavors of the Copy action (for example, Copy Background)
  • edit.paste - for additional flavors of the Paste action (for example, Paste as Link)
  • edit.context - for adding additional actions that apply to the editor in the extension
  • additions - for adding new top level menus to the right of the Perspective menu
  • additions-end - for adding new top level menus to the left of the Window menu

Actions and menus added into these paths will be shown only while the associated editor is active. When the editor is closed, menus and actions will be removed.

Supplied Implementation

The workbench does not provide any built-in content editors, but the end user product may contain some as part of the shipping bundle. In that case, editors will be registered as extensions using the syntax described above.
